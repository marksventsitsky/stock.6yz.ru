## Bitrix24: промо-акции в Сделках (маркетинговая аналитика)

Хранит каталог акций (бренд/город/тип/акция/период) и показывает на карточке
Сделки вкладку **«Акции»** с зависимыми списками: направление (город) → тип
акции → акция. Пока только для сделок в воронке «Продажи» (`CATEGORY_ID=0`).

При сохранении пишет выбор одновременно в два места:
- JSON-снапшот (обычное строковое поле) — для истории/аудита;
- обычные списковые/строковые поля (направление, бренд, тип, название акции) —
  именно по ним строится аналитика/фильтры/воронки в самом Bitrix24, так как
  кастомные JSON-поля отчёты Bitrix не читают.

### Почему вкладка, а не поле-виджет

Регистрация нового ТИПА поля (`userfieldtype.add`) требует более высоких прав
приложения, чем `placement.bind` — на практике первое падало с
`insufficient_scope`, второе — нет. Поэтому вместо кастомного `USERFIELD_TYPE`
акции показываются отдельной вкладкой (`CRM_DEAL_DETAIL_TAB`), а обычные поля
(enum/string) создаются штатным `crm.deal.userfield.add`.

### Возможности

- Вкладка «Акции» на карточке сделки — зависимые списки, можно добавить
  несколько акций сразу.
- Двойная запись при сохранении: JSON-снапшот + обычные CRM-поля.
- Города можно тянуть из существующего в портале Bitrix24 «Списки»
  (`lists.get` / `lists.element.get`) — синхронизируются в локальный
  справочник, чтобы каталог акций не расходился с общим списком городов
  компании.
- Админка каталога — три вкладки: **Каталог** (CRUD, множественный выбор
  городов), **Доступ** (выдать доступ конкретному сотруднику без прав
  администратора портала), **Города** (источник + синхронизация из «Списков»).
  Базовый доступ — проверка `user.admin` по токену портала.

### Локальный запуск

```bash
npm i
cp .env.example .env
npm run seed   # заполняет БД из data/catalog.seed.json (импорт из исходного Excel)
npm run dev
```

Дев-предпросмотр вкладки без Bitrix24: `GET /dev/field-preview?entityType=DEAL&entityId=123`

### Обновление каталога из Excel

`data/catalog.seed.json` — нормализованный экспорт исходного файла с акциями.
После первого запуска редактируйте каталог через админку — Excel больше не
источник истины.

### Регистрация в Bitrix24 (высокоуровнево)

1. Локальное приложение (серверное): путь обработчика — админка сервиса,
   путь установки — `/b24/install`. Права (scope): `crm`, `user`.
2. `client_id`/`client_secret` — в `.env`, переустановить приложение.
3. Зарегистрировать вкладку + поля одним запросом:

```bash
curl -X POST <адрес сервиса>/api/b24/setup \
  -H "x-setup-secret: <SETUP_SECRET>" \
  -H "Content-Type: application/json" \
  -d '{"memberId":"...","domain":"...","accessToken":"..."}'
```

Вызовет `placement.bind` (вкладка «Акции» на сделке), затем
`crm.deal.userfield.add` для 5 полей (JSON-строка + 3 enum + 1 множественное
строковое), с предзаполненными значениями из текущего каталога.

**Если `placement`/`lists` не выбираются в scope-листе локального
приложения** (бывает на self-hosted версиях — это ограничение конкретного
портала, а не наше): создайте **входящий вебхук** (Разработчикам → Другое →
Входящий вебхук) с правами `placement` и `lists`, положите его URL в
`B24_WEBHOOK_URL`. Именно эти два метода (`placement.bind` и `lists.*`)
пойдут через вебхук, всё остальное (`crm.*`, `user.*`) — через обычный
OAuth-токен приложения (`crm`+`user` этому порталу выбрать разрешает).

### Environment

- `PUBLIC_BASE_URL` — публичный адрес сервиса
- `B24_CLIENT_ID` / `B24_CLIENT_SECRET` / `B24_REDIRECT_URI` — локальное приложение Bitrix24
- `SETUP_SECRET` — защищает `/api/b24/setup`
- `ADMIN_SECRET` — резервный доступ к `/api/admin/*` вне Bitrix24 (для отладки)

### Формат JSON-снапшота на сущности

```json
[
  {
    "promoId": "promo-6",
    "brand": "Окошкино",
    "city": "Ростов-на-Дону",
    "type": "Флагманская акция (главная)",
    "title": "Остекли дом или квартиру и получи сплит-систему в подарок",
    "selectedAt": "2026-05-10T12:00:00.000Z"
  }
]
```
