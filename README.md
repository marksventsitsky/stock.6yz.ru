## Bitrix24: Промо-акции в Лидах и Сделках (для маркетинговой аналитики)

Сервис хранит каталог акций (бренд/город/тип/акция/период) и добавляет в карточку
Сделки/Лида вкладку **«Акции»** с зависимыми списками: пользователь выбирает
**направление (город)** → **тип акции** → **акцию**. Устроено "хитро", по
требованию: одновременно с JSON-снапшотом (для истории/аудита) сервис пишет выбор в
**обычные списковые/строковые поля** сделки/лида — именно по ним потом строится
аналитика и фильтры в самом Bitrix24 (по кастомному JSON-полю виджета отчёты Bitrix
строить не умеют).

### Почему вкладка, а не кастомное поле-виджет

Изначально акции планировались как кастомный тип поля (`USERFIELD_TYPE`,
`userfieldtype.add`) — так сделан аналог для рассрочек. На портале bitwork.ru этот
метод (и `placement.bind` тоже) упал с `insufficient_scope`, хотя обычные
`crm.deal.userfield.add`/`crm.lead.userfield.add` тем же токеном (`crm`+`user`)
работали без проблем. Регистрация НОВОГО ТИПА поля требует более высоких прав,
чем создание обычных полей — поэтому вместо поля-виджета акции показываются как
отдельная вкладка на карточке (`placement.bind`, `CRM_DEAL_DETAIL_TAB` /
`CRM_LEAD_DETAIL_TAB`), для чего требуется всего лишь scope `placement` (стандартный,
широко доступный) вместо особого разрешения на кастомные типы полей.

### Что получаете

- **Вкладка «Акции»** на карточке Сделки/Лида — `GET/POST /b24/promo-tab`
  (`placement.bind`, `CRM_DEAL_DETAIL_TAB` / `CRM_LEAD_DETAIL_TAB`). Зависимые
  списки (город → тип → акция), можно добавить несколько акций сразу.
- **Двойная запись** при нажатии «Сохранить в CRM»:
  - `UF_CRM_PROMO_JSON` — полный снапшот выбора (обычное строковое поле, для истории)
  - `UF_CRM_PROMO_CITY`, `UF_CRM_PROMO_BRAND`, `UF_CRM_PROMO_TYPE` — обычные **множественные
    списковые (enumeration)** поля, доступны в фильтрах/отчётах/воронках Bitrix24
  - `UF_CRM_PROMO_NAME` — множественное **строковое** поле (название акции меняется
    каждый месяц, поэтому не enum)
- **Города — из Bitrix24 «Списки»**: в админке есть вкладка «Города», которая находит
  списки портала (`lists.get`) и подтягивает элементы выбранного списка
  (`lists.element.get`) в локальный справочник `city_directory`. Пока источник не
  выбран, город берётся из уже введённых значений в каталоге.
- **Админка каталога акций** — `GET/POST /b24/admin`, доступна как встроенное
  приложение Bitrix24. Три вкладки: **Каталог акций** (CRUD, множественный выбор
  городов вместо свободного текста), **Доступ** (выдать доступ конкретному
  сотруднику без прав администратора портала — поиск через `user.search`), **Города**
  (источник + синхронизация из «Списков»). Основной доступ — проверка `user.admin`
  по токену портала; вкладку «Доступ»/«Города» видят только настоящие админы.

### Локальный запуск

```bash
npm i
cp .env.example .env
npm run seed   # заполняет data.sqlite из data/catalog.seed.json (импорт из Excel)
npm run dev
```

Дев-предпросмотр вкладки без Bitrix24: `GET /dev/field-preview?entityType=DEAL&entityId=123`

### Обновление каталога из Excel

`data/catalog.seed.json` — нормализованный экспорт исходного файла
«Акции Окошкино / Дверной.xlsx» (все листы: Постоянные скидки + помесячные).
После первого запуска редактируйте каталог через `/b24/admin` — Excel больше не
нужен как источник истины.

### Docker (за системным nginx, деплой на сервере)

Репозиторий склонирован прямо на сервер в `/opt/stock.6yz.ru` (см. `restb24.6yz.ru`
для образца этого же паттерна):

```bash
ssh -p 13609 root@80.78.242.60
cd /opt/stock.6yz.ru
git pull
cp .env.example .env   # один раз — затем заполнить реальными значениями
docker compose up -d --build
docker exec b24-stock-promo node dist/scripts/seedCatalog.js   # один раз, при первом деплое
```

nginx-конфиг лежит в `nginx/stock.6yz.ru.conf`, разложен так же, как у
`restb24.6yz.ru`:

```bash
cp /opt/stock.6yz.ru/nginx/stock.6yz.ru.conf /etc/nginx/sites-available/stock.6yz.ru
ln -sf ../sites-available/stock.6yz.ru /etc/nginx/sites-enabled/stock.6yz.ru
nginx -t && systemctl reload nginx
```

HTTPS настроен через certbot (`certonly --webroot -w /var/www/certbot -d stock.6yz.ru`),
сертификат обновляется автоматически.

### Environment

- `PUBLIC_BASE_URL` — публичный адрес сервиса (Bitrix24 должен до него достучаться), напр. `https://stock.6yz.ru`
- `B24_CLIENT_ID` / `B24_CLIENT_SECRET` / `B24_REDIRECT_URI` — для локального приложения Bitrix24
- `SETUP_SECRET` — защищает `/api/b24/setup`
- `ADMIN_SECRET` — резервный доступ к `/api/admin/*` вне Bitrix24 (для отладки); в
  проде основной путь — проверка `user.admin` по токену портала

### Регистрация в Bitrix24 (высокоуровнево)

1. Портал → **Разработчикам → Другое → Локальное приложение**, серверное.
   - Путь обработчика: `https://stock.6yz.ru/b24/admin` (открывается из меню приложений)
   - Путь для первоначальной установки: `https://stock.6yz.ru/b24/install`
   - Права (scope): **`crm`, `user`, `placement`**, а если нужны города из «Списков» — ещё `lists`.
2. Скопировать `client_id`/`client_secret` в `.env`, перезапустить контейнер,
   переустановить приложение — это сохранит `member_id`/токены в `portals`.
3. Зарегистрировать вкладку + поля (обе сущности — сделки и лиды):

```bash
curl -X POST https://stock.6yz.ru/api/b24/setup \
  -H "x-setup-secret: <SETUP_SECRET>" \
  -H "Content-Type: application/json" \
  -d '{"memberId":"...","domain":"your.bitrix24.com","accessToken":"..."}'
```

Это вызовет `placement.bind` (вкладка «Акции» на Сделке и Лиде), затем
`crm.deal.userfield.add`/`crm.lead.userfield.add` для 5 полей (JSON-строка + 3 enum
+ 1 множественное строковое), с предзаполненными значениями города/бренда/типа из
текущего каталога.

**Важно:** если `placement.bind` или регистрация полей падают с `insufficient_scope`,
значит в приложении не хватает нужного scope — добавьте его в настройках
приложения и переустановите (Bitrix должен запросить повторное согласие).

```json
[

    "brand": "Окошкино",
    "city": "Ростов-на-Дону",

    "title": "Остекли дом или квартиру и получи сплит-систему в подарок",
    "selectedAt": "2026-05-10T12:00:00.000Z"
  }
]
```
