## Bitrix24: Промо-акции в Лидах и Сделках (для маркетинговой аналитики)

Сервис хранит каталог акций (бренд/город/тип/акция/период) и даёт в CRM Bitrix24
зависимый виджет выбора акции: пользователь выбирает **направление (город)** →
**тип акции** → **акцию**. Устроено "хитро", по требованию: одновременно с
JSON-снапшотом (для истории/аудита) сервис пишет выбор в **обычные
списковые/строковые поля** сделки/лида — именно по ним потом строится аналитика и
фильтры в самом Bitrix24 (по кастомному JSON-полю виджета отчёты Bitrix строить не умеют).

### Что получаете

- **Поле в Сделке/Лиде** — `GET/POST /b24/userfield/promo`, `USER_TYPE_ID=promo_selector`.
  Зависимые списки (город → тип → акция), можно добавить несколько акций сразу.
- **Двойная запись** при нажатии «Сохранить в CRM»:
  - `UF_CRM_PROMO_JSON` — полный снапшот выбора (для истории)
  - `UF_CRM_PROMO_CITY`, `UF_CRM_PROMO_BRAND`, `UF_CRM_PROMO_TYPE` — обычные **множественные
    списковые (enumeration)** поля, доступны в фильтрах/отчётах/воронках Bitrix24
  - `UF_CRM_PROMO_NAME` — множественное **строковое** поле (название акции меняется
    каждый месяц, поэтому не enum)
- **Админка каталога акций** — `GET/POST /b24/admin`, доступна как встроенное
  приложение Bitrix24 (открывается только администраторам портала — проверка через
  REST `user.admin`). CRUD по акциям + кнопка «Синхронизировать справочники в
  Bitrix» (добавляет новые значения города/бренда/типа в списковые поля, не трогая
  уже существующие ID — прежние сделки/лиды не ломаются).

### Локальный запуск

```bash
npm i
cp .env.example .env
npm run seed   # заполняет data.sqlite из data/catalog.seed.json (импорт из Excel)
npm run dev
```

Дев-предпросмотр виджета без Bitrix24: `GET /dev/field-preview?entityId=CRM_DEAL&entityValueId=123`

### Обновление каталога из Excel

`data/catalog.seed.json` — нормализованный экспорт исходного файла
«Акции Окошкино / Дверной.xlsx» (все листы: Постоянные скидки + помесячные).
После первого запуска редактируйте каталог через `/b24/admin` — Excel больше не
нужен как источник истины.

### Docker (за системным nginx, деплой на сервере)

Репозиторий склонирован прямо на сервер в `/opt/stock.6yz.ru` (см. `restb24.6yz.ru`
для образца этого же паттерна):

```bash
ssh -p 13609 root@80.78.242.60
cd /opt/stock.6yz.ru
git pull
cp .env.example .env   # один раз — затем заполнить реальными значениями
docker compose up -d --build
docker exec b24-stock-promo node dist/scripts/seedCatalog.js   # один раз, при первом деплое
```

nginx-конфиг лежит в `nginx/stock.6yz.ru.conf`, разложен так же, как у
`restb24.6yz.ru`:

```bash
ln -sf /opt/stock.6yz.ru/nginx/stock.6yz.ru.conf /etc/nginx/sites-available/stock.6yz.ru
ln -sf ../sites-available/stock.6yz.ru /etc/nginx/sites-enabled/stock.6yz.ru
nginx -t && systemctl reload nginx
```

Сейчас (2026-07-08) конфиг **только HTTP** — DNS-запись `stock.6yz.ru` ещё не
резолвится ни на публичных резолверах, ни на авторитетных NS Yandex. Как только
она поднимется, выпустить сертификат и переключить на HTTPS:

```bash
certbot certonly --webroot -w /var/www/certbot -d stock.6yz.ru
# затем добавить в конфиг блок `listen 443 ssl` с сертификатом (по образцу restb24.6yz.ru)
# и поменять PUBLIC_BASE_URL / B24_REDIRECT_URI в .env на https://stock.6yz.ru
```

### Environment

- `PUBLIC_BASE_URL` — публичный адрес сервиса (Bitrix24 должен до него достучаться), напр. `https://stock.6yz.ru`
- `B24_CLIENT_ID` / `B24_CLIENT_SECRET` / `B24_REDIRECT_URI` — для локального приложения Bitrix24
- `SETUP_SECRET` — защищает `/api/b24/setup`
- `ADMIN_SECRET` — резервный доступ к `/api/admin/*` вне Bitrix24 (для отладки); в
  проде основной путь — проверка `user.admin` по токену портала

### Регистрация в Bitrix24 (высокоуровнево)

1. Портал → **Разработчикам → Другое → Локальное приложение**, путь обработчика
   установки — любой из `/` (см. `app.get("/")`), права (scope): `crm`, `user`,
   `userfieldtype`.
2. Скопировать `client_id`/`client_secret` в `.env`, перезапустить контейнер,
   переустановить приложение — это сохранит `member_id`/токены в `portals`.
3. Зарегистрировать тип поля + сами поля (обе сущности — сделки и лиды):

```bash
curl -X POST https://stock.6yz.ru/api/b24/setup \
  -H "x-setup-secret: <SETUP_SECRET>" \
  -H "Content-Type: application/json" \
  -d '{"memberId":"...","domain":"your.bitrix24.com","accessToken":"..."}'
```

Это вызовет `userfieldtype.add`, затем `crm.deal.userfield.add` /
`crm.lead.userfield.add` для всех 5 полей (JSON-виджет + 3 enum + 1 строковое), с
предзаполненными значениями города/бренда/типа из текущего каталога.

4. Добавить в карточку сделки/лида новое поле «Выбор акции» (обычным образом через
   настройку полей CRM — Bitrix предложит его в списке типов после шага 3).
5. Добавить в меню приложений Bitrix24 пункт на `/b24/admin` (`placement.bind` с
   `PLACEMENT: "MENU"` либо через настройки приложения) — это и есть админка для
   маркетинга, доступ только администраторам портала.

### Формат JSON-снапшота на сущности

```json
[
  {
    "promoId": "promo-6",
    "brand": "Окошкино",
    "city": "Ростов-на-Дону",
    "type": "Флагманская акция (главная)",
    "title": "Остекли дом или квартиру и получи сплит-систему в подарок",
    "selectedAt": "2026-05-10T12:00:00.000Z"
  }
]
```
